# BOOTSTRAP.md

Этот документ — корневая точка входа для **Bobby Axe** после клонирования репозитория.
Его нужно читать **первым**, до запуска агентов, написания кода и изменения инфраструктуры.

---

## 1. Цель системы

Система строится как оркестратор набора специализированных агентов для подготовки торговых данных, аналитики, генерации сеток и управленческого отчёта.

### Цели в 3–5 пунктах

1. **Собирать и хранить рыночные данные 24/7** по фьючерсным парам USDC с Binance через инфраструктуру `Jack/Data_collector`.
2. **Преобразовывать сырые тики в пригодные для анализа структуры**: секундные агрегаты, OHLC и таймфреймы `1m`, `5m`, `60m`.
3. **Формировать аналитические выводы по каждому тикеру** через `Ben_Kim` на основе набора конфигурируемых стратегий.
4. **Генерировать параметры торговых сеток и распределять капитал** через `Maffi` и `1$_Dollar_Bill`.
5. **Выдавать понятный финальный отчёт пользователю** через `Jusetta`, используя результаты, сохранённые в системе.

---

## 2. Состав агентов

### Оркестратор
- **Bobby Axe** — главный управляющий агент.
  - Следит за подчинёнными агентами.
  - Улучшает их конфигурации, процессы и контракты.
  - Контролирует целостность пайплайна, порядок запуска и Definition of Done.

### Исполнители
- **Jack** — системный администратор и владелец инфраструктуры.
  - Поднимает и обслуживает `Data_collector`.
  - Подключает источники данных.
  - Отвечает за SQL-хранилища, корректность загрузки, хранения и преобразования.

- **Ben_Kim** — аналитик.
  - Принимает тикер, который существует в `Data_collector`.
  - Выполняет анализ по стратегиям и таймфреймам `1m`, `5m`, `60m`.
  - Возвращает краткие заключения для хранения.

- **Maffi** — генератор торговых сеток.
  - Получает аналитические выводы `Ben_Kim`.
  - Суммаризирует сигналы по тикеру.
  - Формирует параметры сетки для **только `5m`**.

- **1$_Dollar_Bill** — аллокатор капитала / риск-менеджер.
  - Получает все сетки по всем тикерам.
  - Анализирует сетки вместе с выводами `Maffi` и `Ben_Kim`.
  - Распределяет условные `100%` капитала между тикерами.

- **Jusetta** — отчётный и визуальный агент.
  - Получает сохранённые результаты.
  - Собирает таблицу.
  - Экспортирует отчёт в PDF и передаёт пользователю.

## 3. Порядок старта

Запуск выполняется **строго последовательно**.
Ни один следующий этап не стартует, пока предыдущий не закрыт по своему Definition of Done.

### Stage 1 — инфраструктура `Jack/Data_collector`
**Запускать первым.**

`Jack` должен:
1. Поднять инфраструктуру сбора данных.
2. Подключить источник Binance.
3. Ограничить universe тикеров фьючерсными парами USDC.
4. Обеспечить live-получение тиков/сделок.
5. Настроить два уровня хранения:
   - сырое tick storage;
   - структурированное агрегированное storage.

### Stage 2 — аналитический пайплайн `Ben_Kim`
**Запускать вторым, только после готовности `Data_collector`.**

`Ben_Kim` должен:
1. Брать только тикеры, реально существующие в хранилище `Jack`.
2. Выполнять анализ по стратегиям.
3. Формировать краткие заключения по таймфреймам `1m`, `5m`, `60m`.
4. Передавать результаты обратно `Jack` на хранение.

### Stage 3 — генерация сеток `Maffi`
**Запускать третьим, только после завершения `Ben_Kim`.**

`Maffi` должен:
1. Забрать аналитические результаты `Ben_Kim`.
2. Суммаризировать сигналы по тикеру.
3. Для каждого тикера сформировать параметры сетки **только для `5m`**.
4. Сохранить результат через `Jack`.

### Stage 4 — аллокация капитала `1$_Dollar_Bill`
**Запускать четвёртым, только после завершения `Maffi`.**

`1$_Dollar_Bill` должен:
1. Получить все готовые сетки по тикерам.
2. Подтянуть заключения `Maffi` и `Ben_Kim`.
3. Рассчитать долю капитала для каждого тикера.
4. Проверить, что сумма аллокаций строго равна `100%`.

### Stage 5 — отчёт `Jusetta`
**Запускать пятой, после того как результаты сохранены и доступны в хранилище.**

`Jusetta` должна:
1. Получить сохранённые результаты из системы.
2. Собрать таблицу по тикерам, аналитике, сеткам и аллокациям.
3. Преобразовать отчёт в PDF.
4. Подготовить финальный артефакт для отправки пользователю.

## 4. Definition of Done по этапам

### DoD: `Jack / Data_collector`
Этап считается завершённым, если:
- источник Binance подключён;
- в intake попадают live-сделки по фьючерсным парам USDC;
- сырые данные сохраняются отдельно по тикерам;
- структурированное хранилище формирует записи уровня `1s`;
- для каждой секунды доступны:
  - `datetime`;
  - `volume` (разница bid/ask за секунду);
  - `O/H/L/C` за секунду;
  - `1m`, `5m`, `60m` OHLC;
  - количество сделок за секунду;
- есть подтверждение, что данные можно прочитать следующему этапу.

### DoD: `Ben_Kim`
Этап считается завершённым, если:
- анализ делается только по валидным тикерам из `Data_collector`;
- для каждого тикера сформированы заключения по стратегиям;
- для каждого заключения указаны:
  - `ticker`;
  - `strategy`;
  - `conclusion`;
  - `time`;
  - `frame` (`1m`, `5m`, `60m`);
- результаты сохранены через `Jack` и доступны для чтения `Maffi`.

### DoD: `Maffi`
Этап считается завершённым, если:
- аналитика `Ben_Kim` успешно прочитана;
- по каждому тикеру выполнена суммаризация сигналов;
- для каждого тикера на `5m` сформированы поля:
  - `direction` (`Long` / `Short`);
  - `grid_upper_price`;
  - `grid_lower_price`;
  - `grid_count`;
  - `sl`;
  - `tp`;
  - короткое объяснение выбора;
- результаты сохранены и доступны для `1$_Dollar_Bill`.

### DoD: `1$_Dollar_Bill`
Этап считается завершённым, если:
- загружены все сетки по всем тикерам;
- для каждой сетки подтянуты обоснования `Maffi` и `Ben_Kim`;
- рассчитана доля капитала по каждому тикеру;
- сумма всех долей равна ровно `100%`;
- результат сохранён для отчётности.

### DoD: `Jusetta`
Этап считается завершённым, если:
- собраны актуальные данные по аналитике, сеткам и аллокации;
- сформирована человекочитаемая таблица;
- таблица экспортирована в PDF;
- финальный отчёт готов к передаче пользователю без ручной доработки.

### DoD: `Bobby Axe`
Оркестрация считается выполненной, если:
- все этапы выполнены в правильном порядке;
- ни один агент не начал работу раньше доступности входных данных;
- артефакты каждого этапа сохранены и доступны следующему агенту;
- зафиксированы источники истины, контракты и runbooks;
- есть понятный список следующих действий после клонирования репозитория.

## 5. Что Bobby Axe должен проверить сразу после клонирования репозитория

Сразу после `git clone` Bobby Axe обязан проверить следующее:

1. **Наличие корневого документа запуска**
   - Убедиться, что этот файл (`BOOTSTRAP.md`) существует и актуален.

2. **Наличие и актуальность ТЗ**
   - Проверить, что `ТЗ.txt` присутствует и не противоречит `BOOTSTRAP.md`.

3. **Структуру источников истины**
   - Убедиться, что понятны пути к:
     - ТЗ;
     - схемам данных;
     - контрактам агентов;
     - runbooks;
     - prompt-файлам.

4. **Пробелы в репозитории**
   - Определить, какие критически важные артефакты ещё отсутствуют.
   - Минимум зафиксировать missing items, если нет схем БД, контрактов или runbooks.

5. **Порядок оркестрации**
   - Проверить, что последовательность `Jack -> Ben_Kim -> Maffi -> 1$_Dollar_Bill -> Jusetta` явно описана.

6. **Готовность к запуску инфраструктуры**
   - Проверить, есть ли конфиги окружения, описания сервисов, параметры подключения к Binance и SQL.

7. **Контракты входов/выходов между агентами**
   - Проверить, описаны ли обязательные поля для передачи данных между этапами.

8. **Runbooks и аварийные инструкции**
   - Проверить, есть ли инструкции на случай падения ingest, нарушения схемы данных или несходимости аллокации.

Если чего-то из перечисленного нет, Bobby Axe не должен гадать: он должен сначала оформить недостающие документы/контракты, а потом запускать агентов.

## 6. Источники истины (Source of Truth)

При конфликте документов приоритет должен быть следующим:

1. **ТЗ / product intent**
   - `ТЗ.txt`
   - Это первичный бизнес-источник по ролям агентов и общему смыслу системы.

2. **Корневой документ запуска**
   - `BOOTSTRAP.md`
   - Это источник истины по очередности старта, operational expectations и post-clone действиям.

3. **Схемы данных**
   - Канонический путь должен быть: `docs/data-schemas/`
   - Здесь должны жить описания raw tick storage, aggregated storage, аналитических результатов, сеток и аллокаций.

4. **Контракты агентов**
   - Канонический путь должен быть: `docs/agent-contracts/`
   - Здесь должны быть зафиксированы input/output-контракты для:
     - `Jack`
     - `Ben_Kim`
     - `Maffi`
     - `1$_Dollar_Bill`
     - `Jusetta`
     - `Bobby Axe`

5. **Runbooks**
   - Канонический путь должен быть: `docs/runbooks/`
   - Здесь должны описываться запуск, восстановление, проверки, алерты и ручные процедуры.

6. **Prompt-файлы**
   - Канонический путь должен быть: `prompts/`
   - Здесь должны храниться системные и рабочие промпты для каждого агента.

### Правило приоритета
Если содержимое расходится:
- бизнес-смысл берётся из `ТЗ.txt`;
- порядок запуска и действия после клонирования — из `BOOTSTRAP.md`;
- точные поля данных — из схем данных;
- правила обмена между агентами — из agent contracts;
- порядок эксплуатации и восстановления — из runbooks;
- формулировки поведения агентов — из prompt-файлов.

Если нужный источник истины отсутствует, Bobby Axe должен трактовать это как **blocker**, а не как приглашение импровизировать.
