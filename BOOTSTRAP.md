# BOOTSTRAP.md

Этот документ — корневая точка входа для **Bobby Axe** после клонирования репозитория.
Его нужно читать **первым**, до запуска агентов, написания кода и изменения инфраструктуры.

---

## 1. Цель системы

Система строится как оркестратор набора специализированных агентов для подготовки торговых данных, аналитики, генерации сеток и управленческого отчёта.

### Цели в 3–5 пунктах

1. **Собирать и хранить рыночные данные 24/7** по фьючерсным парам USDC с Binance через инфраструктуру `Jack/Data_collector`.
2. **Преобразовывать сырые тики в пригодные для анализа структуры**: секундные агрегаты, OHLC и таймфреймы `1m`, `5m`, `60m`.
3. **Формировать аналитические выводы по каждому тикеру** через `Ben_Kim` на основе набора конфигурируемых стратегий.
4. **Генерировать параметры торговых сеток и распределять капитал** через `Maffi` и `1$_Dollar_Bill`.
5. **Выдавать ранний preview отдельно от финального пользовательского отчёта**: черновик допускается после `Ben_Kim`, а финальный `report_job`/PDF — только после аллокаций.

## 2. Состав агентов

### Оркестратор
- **Bobby Axe** — главный управляющий агент.
  - Следит за подчинёнными агентами.
  - Улучшает их конфигурации, процессы и контракты.
  - Контролирует целостность пайплайна, порядок запуска и Definition of Done.

### Исполнители
- **Jack** — системный администратор и владелец инфраструктуры.
  - Поднимает и обслуживает `Data_collector`.
  - Подключает источники данных.
  - Отвечает за SQL-хранилища, корректность загрузки, хранения и преобразования.

- **Ben_Kim** — аналитик.
  - Принимает тикер, который существует в `Data_collector`.
  - Выполняет анализ по стратегиям и таймфреймам `1m`, `5m`, `60m`.
  - Возвращает analysis-only результаты для хранения.

- **Maffi** — генератор торговых сеток.
  - Получает аналитические выводы `Ben_Kim`.
  - Суммаризирует сигналы по тикеру.
  - Формирует параметры сетки для **только `5m`**.

- **1$_Dollar_Bill** — аллокатор капитала / риск-менеджер.
  - Получает все сетки по всем тикерам.
  - Анализирует сетки вместе с выводами `Maffi` и `Ben_Kim`.
  - Распределяет условные `100%` капитала между тикерами.

- **Jusetta** — отчётный и визуальный агент.
  - Может собрать ранний `analysis_preview` только на основе analysis-only результатов `Ben_Kim`.
  - Собирает финальный `report_job`, таблицу и PDF только после завершения `Maffi` и `1$_Dollar_Bill`.
  - Не имеет права выдавать финальный user-facing package без сохранённых аллокаций.

## 3. Канонический порядок стадий

Запуск выполняется **строго последовательно**.
Ни один следующий этап не стартует, пока предыдущий не закрыт по своему Definition of Done.

### Stage 0 — инфраструктура `Jack/Data_collector`
**Запускать первым.**

`Jack` должен:
1. Поднять инфраструктуру сбора данных.
2. Подключить источник Binance.
3. Ограничить universe тикеров фьючерсными парами USDC.
4. Обеспечить live-получение тиков/сделок.
5. Настроить два уровня хранения:
   - сырое tick storage;
   - структурированное агрегированное storage.

### Stage 1 — analysis-only пайплайн `Ben_Kim`
**Запускать вторым, только после готовности `Data_collector`.**

`Ben_Kim` должен:
1. Брать только тикеры, реально существующие в хранилище `Jack`.
2. Выполнять анализ по стратегиям.
3. Формировать краткие заключения по таймфреймам `1m`, `5m`, `60m`.
4. Передавать analysis-only результаты обратно `Jack` на хранение.

### Stage 2 — опциональный ранний preview `analysis_preview`
**Необязательный этап. Запускать только после завершения `Ben_Kim`.**

`Jusetta` может:
1. Взять только analysis-only результаты `Ben_Kim`.
2. Собрать черновик `analysis_preview`.
3. Явно пометить артефакт как preview/черновик.
4. Не называть этот артефакт `report_job`, финальным PDF или финальным пользовательским отчётом.

### Stage 3 — генерация сеток `Maffi`
**Запускать после `Ben_Kim` независимо от того, выпускался ли preview.**

`Maffi` должен:
1. Забрать аналитические результаты `Ben_Kim`.
2. Суммаризировать сигналы по тикеру.
3. Для каждого тикера сформировать параметры сетки **только для `5m`**.
4. Сохранить результат через `Jack`.

### Stage 4 — аллокация капитала `1$_Dollar_Bill`
**Запускать только после завершения `Maffi`.**

`1$_Dollar_Bill` должен:
1. Получить все готовые сетки по тикерам.
2. Подтянуть заключения `Maffi` и `Ben_Kim`.
3. Рассчитать долю капитала для каждого тикера.
4. Проверить, что сумма аллокаций строго равна `100%`.

### Stage 5 — финальный отчёт `Jusetta`
**Запускать последней и только после завершения `Maffi` и `1$_Dollar_Bill`.**

`Jusetta` должна:
1. Получить сохранённые результаты аналитики, сеток и аллокаций.
2. Собрать таблицу по тикерам, аналитике, сеткам и аллокациям.
3. Преобразовать отчёт в PDF.
4. Подготовить финальный `report_job` и пользовательский пакет.

**Жёсткое правило:** без сохранённых аллокаций запрещены:
- `report_job`;
- финальный пользовательский PDF;
- любой user-facing report package, который выглядит как окончательный выпуск.

## 3A. Realtime snapshot-run по кнопке

Для пользовательского realtime-запуска по кнопке Bobby Axe обязан завести отдельную orchestration-сущность `snapshot_run` **до** старта `Jack`, `Ben_Kim`, `Maffi`, `1$_Dollar_Bill` и `Jusetta`.

### Обязательные правила snapshot-run
- `snapshot_run` создаётся orchestration control plane под управлением `Bobby Axe`; пользователь или UI только инициируют запрос, но не создают run напрямую.
- `correlation_id` фиксируется при создании `snapshot_run` и затем используется всеми stage-артефактами этого запуска без изменений.
- В момент старта фиксируется `frozen_universe` — упорядоченный список тикеров, доступных для realtime-run на стороне `Jack`; дальше список тикеров внутри run менять нельзя.
- `as_of_utc` фиксируется как последний общий безопасный UTC-срез, для которого `Jack` уже materialized данные по всему frozen universe; новые данные, пришедшие после этого времени, в текущий run не включаются.
- `Ben_Kim` читает только run-scoped frozen market snapshot до `as_of_utc` и строит заключения `1m/5m/60m` по тикерам из frozen universe.
- `Maffi` читает только usable `analysis_result` текущего `correlation_id` и строит `grid_proposal` только для `5m` по frozen universe.
- `1$_Dollar_Bill` читает только frozen universe, зафиксированный в начале snapshot-run, плюс run-scoped результаты `Ben_Kim` и `Maffi`; ему запрещено пересобирать universe по «текущему состоянию» базы во время расчёта.
- Сумма аллокаций `1$_Dollar_Bill` должна быть `100%` **по frozen universe snapshot-run**, иначе результат недетерминирован и этап не принимается.
- Run считается завершённым только когда весь frozen universe дошёл до terminal outcome, а для publishable-выпуска дополнительно готов `report_job` того же `correlation_id`, `as_of_utc` и frozen universe.
- Переиспользовать между повторными попытками можно raw/curated market data на фиксированном `as_of_utc`; `Maffi`, `1$_Dollar_Bill` и финальный `report_job` обязаны пересчитываться заново для каждого нового пользовательского snapshot-run.

## 5. Definition of Done по этапам

### DoD: `Jack / Data_collector`
Этап считается завершённым, если:
- источник Binance подключён;
- в intake попадают live-сделки по фьючерсным парам USDC;
- сырые данные сохраняются отдельно по тикерам;
- структурированное хранилище формирует записи уровня `1s`;
- для каждой секунды доступны:
  - `datetime`;
  - `volume` (разница bid/ask за секунду);
  - `O/H/L/C` за секунду;
  - `1m`, `5m`, `60m` OHLC;
  - количество сделок за секунду;
- есть подтверждение, что данные можно прочитать следующему этапу.

### DoD: `Ben_Kim`
Этап считается завершённым, если:
- анализ делается только по валидным тикерам из `Data_collector`;
- для каждого тикера сформированы заключения по стратегиям;
- для каждого заключения указаны:
  - `ticker`;
  - `strategy`;
  - `conclusion`;
  - `time`;
  - `frame` (`1m`, `5m`, `60m`);
- результаты сохранены через `Jack` и доступны для чтения `Maffi` и, при необходимости, для `analysis_preview`.

### DoD: `analysis_preview`
Этап считается завершённым, если:
- preview построен только на analysis-only результатах `Ben_Kim`;
- артефакт явно помечен как `analysis_preview` или draft;
- в preview нет утверждений, что это финальный `report_job` или финальный пользовательский PDF;
- preview отделён от финального user-facing package по имени, статусу и пути хранения.

### DoD: `Maffi`
Этап считается завершённым, если:
- аналитика `Ben_Kim` успешно прочитана;
- по каждому тикеру выполнена суммаризация сигналов;
- для каждого тикера на `5m` сформированы поля:
  - `direction` (`Long` / `Short`);
  - `grid_upper_price`;
  - `grid_lower_price`;
  - `grid_count`;
  - `sl`;
  - `tp`;
  - короткое объяснение выбора;
- результаты сохранены и доступны для `1$_Dollar_Bill`.

### DoD: `1$_Dollar_Bill`
Этап считается завершённым, если:
- загружены все сетки по всем тикерам;
- для каждой сетки подтянуты обоснования `Maffi` и `Ben_Kim`;
- рассчитана доля капитала по каждому тикеру;
- сумма всех долей равна ровно `100%`;
- результат сохранён для отчётности и разблокирует финальный stage `Jusetta`.

### DoD: `Jusetta`
Этап считается завершённым, если:
- для финального отчёта собраны актуальные данные по аналитике, сеткам и аллокации;
- сформирована человекочитаемая таблица;
- таблица экспортирована в PDF;
- `report_job` и PDF выпущены только после наличия сохранённых аллокаций;
- финальный отчёт готов к передаче пользователю без ручной доработки.

### DoD: `Bobby Axe`
Оркестрация считается выполненной, если:
- все этапы выполнены в правильном порядке;
- ранний `analysis_preview` чётко отделён от финального `report_job`;
- ни один агент не начал работу раньше доступности входных данных;
- финальные пользовательские артефакты не выпущены без аллокаций;
- артефакты каждого этапа сохранены и доступны следующему агенту;
- зафиксированы источники истины, контракты и runbooks;
- есть понятный список следующих действий после клонирования репозитория.

     - `Jack`

   - Канонический путь должен быть: `docs/runbooks/`

   - Канонический путь должен быть: `prompts/`
