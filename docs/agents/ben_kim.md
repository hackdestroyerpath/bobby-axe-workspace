# Ben Kim — стандарт ответа

Этот документ задаёт канонический формат ответа аналитического агента `Ben_Kim`, который Bobby Axe может использовать как эталон при интеграции, хранении и последующей маршрутизации результатов.

## Назначение

`Ben_Kim` возвращает один нормализованный объект ответа на одну комбинацию:

- `symbol`
- `timeframe`
- `strategy_id`

Документ дополняет контракт `contracts/ben_kim.md` и делает акцент на человекочитаемом кратком ответе, который можно валидировать и сохранять в downstream-пайплайне.

## Стандарт полей ответа

| Поле | Тип | Обязательное | Описание |
| --- | --- | --- | --- |
| `symbol` | `string` | Да | Канонический тикер, например `BTCUSDC`. |
| `timeframe` | `string` | Да | Таймфрейм анализа. Допустимые значения: `1m`, `5m`, `60m`. |
| `strategy_id` | `string` | Да | Идентификатор стратегии из `config/ben_kim_strategies.yaml`. |
| `status` | `string` | Да | Бизнес-статус результата. Допустимые значения: `ready`, `partial`, `rejected`. |
| `result_code` | `string` | Да | Технический итог выполнения. Допустимые значения: `ok`, `skipped`, `insufficient_data`, `error`. |
| `summary` | `string` | Да | Краткое заключение стратегии. Должно быть конкретным, без воды, и не длиннее лимита `max_conclusion_length` из конфига стратегии. |
| `confidence` | `number` | Да | Уверенность модели в диапазоне от `0` до `1`. Для `partial` и `rejected` допускается `0`. |
| `generated_at_utc` | `string` | Да | Время генерации ответа в формате ISO 8601 UTC, например `2026-03-21T12:05:00Z`. |

## Правила заполнения

### 1. `symbol`
- Только один тикер на один ответ.
- Формат должен совпадать с тикером из `Data_collector`.

### 2. `timeframe`
- Должен соответствовать реально выполненному анализу.
- Если стратегия не поддерживает таймфрейм, ответ должен иметь `status=partial` и `result_code=skipped`.

### 3. `strategy_id`
- Должен точно совпадать со значением `strategy_id` из конфига.
- `display_name` в ответ не дублируется, чтобы payload оставался компактным.

### 4. `status`
Используется один из трёх бизнес-статусов:

- `ready` — результат можно использовать downstream как валидную аналитику.
- `partial` — результат можно сохранить, показать в отчётах и использовать для диагностики, но нельзя пускать в торговое решение.
- `rejected` — результат отклонён и может использоваться только для аудита, алертов и технического разбора.

### 5. `result_code`
Используется один из четырёх технических итогов:

- `ok` — стратегия отработала успешно и вернула осмысленное заключение.
- `skipped` — стратегия сознательно не запускалась или прервала анализ по одному из `skipped_conditions`.
- `insufficient_data` — для вывода не хватило входных данных, истории или индикаторов.
- `error` — выполнение завершилось технической ошибкой, и результат нельзя считать валидным сигналом.

### 6. Обязательное соответствие `status` и `result_code`
- `ready` ↔ `ok`
- `partial` ↔ `skipped` или `insufficient_data`
- `rejected` ↔ `error`

### 7. `summary`
- Одно короткое заключение без вложенных структур.
- Если `status=ready` и `result_code=ok`, текст должен содержать суть наблюдения и предполагаемое действие/контекст.
- Если `status=partial`, текст должен кратко объяснять причину пропуска или нехватки данных.
- Если `status=rejected`, текст должен быть пригоден для логов и операторской диагностики.

### 8. `confidence`
- Значение от `0.00` до `1.00`.
- Для `ready` обычно больше `0`, если стратегия действительно пришла к выводу.
- Для `partial` и `rejected` допустимо `0`.

### 9. `generated_at_utc`
- Всегда в UTC.
- Всегда в формате ISO 8601 с суффиксом `Z`.
- Отражает момент финализации ответа, а не время рыночного события.

## Рекомендуемый JSON-шаблон

```json
{
  "symbol": "BTCUSDC",
  "timeframe": "5m",
  "strategy_id": "momentum_bias_v1",
  "status": "ready",
  "result_code": "ok",
  "summary": "Пятиминутный импульс направлен вверх: цена удерживает локальный диапазон и объём поддерживает продолжение движения.",
  "confidence": 0.78,
  "generated_at_utc": "2026-03-21T12:05:00Z"
}
```

## Эталонные примеры для Bobby Axe

Ниже приведены эталонные ответы на одном тикере `BTCUSDC`, которые Bobby Axe может использовать как reference-output для проверки формата и статусов.

### Пример 1 — успешный ответ

```json
{
  "symbol": "BTCUSDC",
  "timeframe": "5m",
  "strategy_id": "momentum_bias_v1",
  "status": "ready",
  "result_code": "ok",
  "summary": "На 5m сохраняется умеренный восходящий импульс: close держится выше локальной базы, а объём не подтверждает немедленное истощение движения.",
  "confidence": 0.81,

}
```

## Практические заметки для Bobby Axe

- Сначала проверять, что `strategy_id` существует в `config/ben_kim_strategies.yaml`.
- Затем валидировать `timeframe`, `status` и `result_code` как связанную пару.
- После этого проверять длину `summary` относительно `max_conclusion_length` соответствующей стратегии.
- Только ответы с `status=ready` и `result_code=ok` являются usable-аналитикой.
- Ответы со статусами `partial` и `rejected` можно хранить и маршрутизировать в логи/отчёты, но нельзя интерпретировать как торговый сигнал без отдельной бизнес-логики.
