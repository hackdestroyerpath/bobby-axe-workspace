# PING_5M_REGULATION.md — Регламент пайплайна команды Ping_5m

## 1. Назначение документа
Этот документ фиксирует единый рабочий регламент команды для пайплайна `Ping_5m`.

Документ обязателен для:
- Bobby Axelrod
- Jack
- Ben_Kim
- Maffi
- Dollar_Bill

Цель регламента:
- убрать разночтения;
- закрепить ownership;
- сделать pipeline понятным для исполнения и проверки;
- задать единый источник истины по данным и статусам.

---

## 2. Scope
Текущий регламент описывает только MVP-контур:
- команда: `Ping_5m`
- основной рабочий таймфрейм: `5m`
- анализ со связанными старшим и младшим ТФ

Другие рабочие ТФ в текущий регламент не входят и будут добавлены позже отдельным расширением.

---

## 3. Основные понятия
### Таймфреймы
- `TF` — основной рабочий таймфрейм
- `senior_TF` — старший таймфрейм
- `junior_TF` — младший таймфрейм

### Таблица соответствия
Для текущего MVP:
- если `TF = 5m`, тогда:
  - `senior_TF = 60m`
  - `junior_TF = 1m`

### Ключевые сущности
- `ping_id` — единый идентификатор конкретного пинга
- `ticker` — тикер/монета, входящая в анализ
- `strategy` — одна аналитическая стратегия Ben_Kim
- `artifact` — любой формализованный объект, сохраненный в SQL через API

---

## 4. Роли и ownership
### Bobby Axelrod
Отвечает за:
- запуск команды `Ping_5m`;
- согласование списка тикеров с Boss;
- генерацию `ping_id`;
- регистрацию пинга;
- запуск аналитического контура;
- контроль completion criteria;
- финальный доклад о завершении.

### Jack
Отвечает за:
- SQL-хранилище;
- API-доступ на чтение/запись;
- таблицы pipeline;
- технические логи SQL-вызовов;
- корректность хранения и статусов пинга.

### Ben_Kim
Отвечает за:
- стратегическую аналитику по тикеру;
- 12 strategy-level отчетов на тикер;
- 3 total-TF отчета на тикер;
- запись результатов в SQL.

### Maffi
Отвечает за:
- grid-анализ на рабочем TF;
- 1 итоговый grid-object на тикер;
- запись результата в SQL.

### Dollar_Bill
Отвечает за:
- итоговую оценку тикера на основе полного набора артефактов;
- расчет `GRID_pnl`;
- выпуск `DB_report`;
- запись результата в SQL.

---

## 5. Source of truth
После регистрации `ping_id` в таблице `Pings` именно SQL-хранилище Jack является единственным source of truth.

Это значит:
- список тикеров читается из SQL;
- статус пинга читается из SQL;
- готовность стадий проверяется по SQL;
- сообщения в чате не заменяют зарегистрированные данные в хранилище.

---

## 6. Таблицы SQL
Все таблицы хранятся в контуре Jack.

Чтение и запись — строго через API.

### Основные таблицы
- `Pings` — история и состояние пингов
- `Ticks` — коллектор с тиками
- `BK_per_strategy_report` — отчеты Ben_Kim по каждой стратегии
- `BK_total_report` — итоговые отчеты Ben_Kim по каждому TF
- `Maffi_grids_report` — итоговые grid-отчеты Maffi
- `DB_report` — итоговые отчеты Dollar_Bill
- `SQL_Logs` — технический журнал SQL API-операций

### Требование к `SQL_Logs`
Jack обязан вести отдельную таблицу `SQL_Logs`, где должны быть отцифрованы:
- все GET/POST запросы;
- таблица назначения;
- ключ/идентификатор сущности;
- время запроса;
- результат вызова;
- технический статус.

---

## 7. Статусная модель пинга
Каждый `ping_id` должен иметь явный статус.

Рекомендуемая модель статусов:
- `created` — ping зарегистрирован
- `running` — Ben_Kim и/или Maffi выполняют свои стадии
- `waiting_for_db_stage` — Ben_Kim и Maffi завершили работу, ожидается/выполняется stage Dollar_Bill
- `completed` — все тикеры по ping_id полностью завершены
- `failed` — pipeline завершился с критической ошибкой
- `partial_failed` — часть тикеров не прошла pipeline полностью

Bobby не объявляет пинг завершенным до тех пор, пока source of truth не подтверждает корректный конечный статус.

---

## 8. Pipeline команды Ping_5m
### Stage 1 — Инициация Bobby
1. Boss дает Bobby команду `Ping_5m`.
2. Bobby получает из коллектора Jack доступную номенклатуру тикеров.
3. Bobby согласовывает с Boss, какие тикеры входят в текущий пинг.
4. Bobby генерирует единый `ping_id` для всего набора тикеров.
5. Bobby регистрирует `ping_id` в таблице `Pings`.
6. В `Pings` должны быть зафиксированы:
   - `ping_id`
   - список согласованных тикеров
   - рабочий `TF`
   - время создания
   - статус `created`
7. После регистрации Bobby передает `ping_id` Ben_Kim и Maffi.
8. Этот момент является формальным стартом pipeline.

После регистрации `Pings` статус переводится в `running`.

---

## 9. Stage Ben_Kim
### Вход
Ben_Kim читает из `Pings`:
- `ping_id`
- набор тикеров для анализа
- рабочий `TF`

### Логика работы
Для каждого тикера Ben_Kim:
1. Запускает 12 аналитических единиц:
   - 4 стратегии × 3 таймфрейма (`TF`, `senior_TF`, `junior_TF`)
2. Каждая аналитическая единица:
   - использует свой алгоритм подготовки данных;
   - читает данные из коллектора тиков;
   - формирует запрос в LLM;
   - получает strategy-level textual output.

### Выход первого слоя
На каждый тикер создаются 12 объектов в `BK_per_strategy_report`.

Каждый объект должен содержать:
- `ticker`
- `tf`
- `strategy_name`
- `strategy_text`
- `ping_id`
- `created_at`
- `id`

### Выход второго слоя
После этого Ben_Kim для каждого тикера собирает по каждому TF стек из 4 strategy-level отчетов и делает 1 итоговый LLM-запрос на каждый TF.

Итог:
- 3 total-report объекта на тикер

Каждый объект в `BK_total_report` должен содержать:
- `ticker`
- `tf`
- `strategy_report`
- `ping_id`
- `created_at`
- `id`

### Норма выхода Ben_Kim
На 1 тикер:
- 12 объектов в `BK_per_strategy_report`
- 3 объекта в `BK_total_report`

После завершения Ben_Kim записывает все результаты в SQL через API.

---

## 10. Stage Maffi
### Вход
Maffi читает из `Pings`:
- `ping_id`
- набор тикеров для анализа
- рабочий `TF`

### Логика работы
Для каждого тикера Maffi:
1. Собирает аналитические данные со своих скриптов.
2. Подготавливает input для LLM:
   - `ticker`
   - `algo_data`
   - `time`
   - `prompt`
   - `TF`
3. Делает 1 LLM-запрос на тикер.

### Выход
На каждый тикер создается 1 объект в `Maffi_grids_report`.

Каждый объект должен содержать:
- `ticker`
- `tf` (рабочий TF)
- `grid_parameters`
- `grid_report`
- `ping_id`
- `created_at`
- `id`

### Состав `grid_parameters`
Как минимум должны быть:
- направление (`long` или `short`)
- верхняя цена
- нижняя цена
- stop loss
- take profit
- количество сеток

После завершения Maffi записывает результаты в SQL через API.

---

## 11. Trigger и stage Dollar_Bill
### Условие запуска
Stage Dollar_Bill должен запускаться только после того, как по каждому тикеру текущего `ping_id` в source of truth присутствует полный комплект артефактов:
- 12 объектов `BK_per_strategy_report`
- 3 объекта `BK_total_report`
- 1 объект `Maffi_grids_report`

Только после этого pipeline переходит в статус:
- `waiting_for_db_stage`

И инициируется stage Dollar_Bill с тем же `ping_id`.

### Логика работы Dollar_Bill
Для каждого тикера Dollar_Bill:
1. Загружает все связанные с этим `ping_id` артефакты тикера:
   - 12 strategy-level объектов Ben_Kim
   - 3 total-report объекта Ben_Kim
   - 1 grid-object Maffi
2. Делает 1 итоговый LLM-запрос на тикер.

В запросе передаются:
- `ticker`
- `time`
- рабочий `TF`
- данные всех 16 объектов

### Выход Dollar_Bill
На 1 тикер создается 1 объект в `DB_report`.

Объект должен содержать:
- `ticker`
- `tf` (рабочий TF)
- `grid_pnl`
- `bill_report`
- `ping_id`
- `created_at`
- `id`

Где:
- `grid_pnl` — числовая процентная оценка
- `bill_report` — краткая текстовая оценка

После завершения Dollar_Bill записывает результат в `DB_report` через API.

---

## 12. Completion criteria
Pipeline `Ping_5m` считается завершенным только если одновременно выполнены все условия:

1. `ping_id` зарегистрирован в `Pings`
2. все тикеры из `Pings` обработаны Ben_Kim
3. все тикеры из `Pings` обработаны Maffi
4. по каждому тикеру есть полный комплект Ben_Kim + Maffi артефактов
5. по каждому тикеру создан объект в `DB_report`
6. статус пинга в source of truth переведен в `completed`
7. после этого Bobby делает финальный доклад о завершении

Если хотя бы один тикер не дошел до `DB_report`, пинг не считается fully completed.

---

## 13. Done criteria по ролям
### Bobby — done, если:
- тикеры согласованы
- `ping_id` создан
- `Pings` зарегистрирован
- source of truth подтверждает завершение
- финальный доклад сделан

### Jack — done, если:
- таблицы доступны через API
- все стадии корректно читают/пишут данные
- `SQL_Logs` фиксирует обращения
- статусы пинга отражены корректно

### Ben_Kim — done, если:
- на каждый тикер создано 12 strategy-level отчетов
- на каждый тикер создано 3 total-report объекта
- все записано в SQL

### Maffi — done, если:
- на каждый тикер создан 1 grid-object
- все записано в SQL

### Dollar_Bill — done, если:
- на каждый тикер создан 1 `DB_report`
- `grid_pnl` и `bill_report` записаны в SQL

---

## 14. Дедупликация и идемпотентность
Для pipeline должны действовать следующие правила:

1. Один и тот же `ping_id` не должен порождать дубли артефактов при повторном запуске стадии.
2. Каждая запись должна быть однозначно связана с:
   - `ping_id`
   - `ticker`
   - `tf`
   - типом stage
   - при необходимости `strategy_name`
3. Повторный запуск stage должен:
   - либо обновлять/перезаписывать существующий артефакт по тем же ключам;
   - либо быть явно запрещен, если запись уже существует.
4. Jack обязан не допускать silent-duplicate writes на уровне API/хранилища.

---

## 15. Error policy
### Общий принцип
Ошибки должны быть формализованы, а не скрыты.

### Hard fail
Переход в `failed` допустим, если:
- SQL/API недоступен критически
- pipeline не может продолжаться
- source of truth не может быть обновлен

### Partial fail
Переход в `partial_failed` допустим, если:
- часть тикеров завершена,
- но хотя бы один тикер не дошел до полного набора артефактов
- или не дошел до `DB_report`

### Обязательные правила
1. Нельзя объявлять ping завершенным при неполном наборе артефактов.
2. Если одна стадия падает по одному тикеру, это должно быть явно отражено в статусе.
3. Если LLM вернул невалидный ответ, stage обязан зафиксировать ошибку, а не молча пропустить тикер.
4. Если SQL write не прошел, это должно фиксироваться как техническая ошибка, а не считаться done.
5. Bobby докладывает `blocked`, `failed` или `partial_failed`, если source of truth не подтверждает completion.

---

## 16. Операционный принцип для команды
Команда работает не по чату и не по “понятно из контекста”, а по следующему правилу:
- сначала зарегистрированный `ping_id`
- потом pipeline по source of truth
- потом запись артефактов
- потом проверка completion criteria
- потом финальный доклад Bobby

Любой участник команды обязан ориентироваться на:
- SQL как source of truth
- свой четкий stage
- свой обязательный artifact
- завершение только по формальным признакам

---

## 17. MVP-границы текущей версии
В текущую версию регламента входят:
- только `Ping_5m`
- только рабочий `TF = 5m`
- `senior_TF = 60m`
- `junior_TF = 1m`
- полный pipeline Bobby → Jack → Ben_Kim + Maffi → Dollar_Bill → Bobby

В текущую версию пока не входят:
- другие рабочие ТФ
- расширенные схемы multi-ping orchestration
- дополнительные decision layers сверх описанного контура

---

## 18. Финальное правило
Пинг считается завершенным не тогда, когда кому-то кажется, что “анализ уже сделали”, а тогда, когда:
- весь набор тикеров по `ping_id` прошел все стадии,
- все обязательные артефакты записаны,
- статус в source of truth переведен в `completed`,
- и только после этого Bobby делает финальный управленческий доклад.
