1) Jack создаст машину, которую будет потом обслуживать:
- Data_collector - машина, которая собирает данные.
- Работает 24/7. На вход принимает данные, затем хранит, преобразовывает, передает.
Джек будет подключать к ней источники данных и следить, что бы данные корректно получались, хранились и преобразовывались.
Джек поднимет SQL и будет хранить там все входящие данные, затем преобразовывать при необходимости и выдавать.
На вход к датаколлектору подключается источник данных (дается имя). В нашем случае данные будут с бинанс, а тикеры все пары фьючерсов USDC.
Для начала Data_collector будет в лайве получать тики (сделки) и хранить их (отдельное хранилище SQL). По каждому тикеру.
Каждый тикер нужно будет преобразовывать, например (Другое хранилище):
- по ним строить OHLC (1m),(5m),(60m), считать и хранить рядом для каждого тикера. То есть создать колонки  (1m),(5m),(60m), * O,H,L,C

То есть в одно sql хранилище заходит только в режиме лайв связка "volume-bid/ask-time" ДЛЯ ОДНОГО ТИКЕРА.(1 sql хранилище на 1 тикер).

А второе достает уже от туда данные, и структурирует (что бы потом уже удобно было просто достать). Тоже 1 sql хранилище на 1 тикер (итого, 2sql на 1 тикер)

Во втором sql должны быть для каждой строки (row = 1s):
-дата время
-volume (раница bid ask за секунду)
-O,H,L,C (за секунду)
-(1m),(5m),(60m), * O,H,L,C
-Общее кол-во сделоку за сек.

2) Ben_Kim. По запросу будет делать аналитику. На вход будет принимать Тикер, который есть в Data_collector (иначе нет).
На выходе будет давать своё КОРОТКОЕ заключение по каждой из стратегии (набор стратегий конфигурируется в настройках АИ):
- Ценовые уровни
- Уровни Фибоначи
- Волны Элиота
- Вертикальне объему
- Активность сделок
- Горизонтальные объемы
- Новостной фон (политика, экономика)
- корелляция с BTC
- активность сделок (кол-во в сек)
- RSI + MACD
- Каналы

По 3 заклчения ( (1m),(5m),(60m) на КАЖДЫЙ тикер).

То есть на выходе Бен Ким даёт данные, где будет расписан каждый тикер,который он получил, в разрезе
стратегий, которые он рассматривает (КАЖДАЯ СТРАТЕГИЯ * КАЖДЫЙ ТИКЕР)
Т.е. тикер - стратегия (ценовые уровни, например) - заключение (нет близких уровней, например).
Стратегию можно игнорировать, если на то основание.

На выходе получаем.

Тикер - стратегия - заключение - время  - фрейм (1,5,60).

Это analysis-only артефакт. Он является промежуточным и сохраняется Джеку на хранение.

2.1) После завершения Ben_Kim можно опционально выпустить ранний черновик `analysis_preview`.
`analysis_preview`:
- строится только на analysis-only артефактах Ben_Kim;
- является промежуточным черновиком, а не финальным отчётом;
- может быть отправлен пользователю только если явно помечен как preview/черновик;
- не может называться `report_job`, пользовательским PDF или финальным report package;
- не требует аллокаций и не заменяет финальный отчёт.

Для каждой Тикер - фрейм, маффи возвращает:

этих данныех, для КАЖДОГО тикера, ТОЛЬКО 5м дает параметры сетки:

+ Заключениие (короткое) , почему выбрал такие параметры (на основании данных Бенкима).

Тоже закидываем Джеку на хранение.
Это тоже промежуточный артефакт, а не пользовательский финальный отчёт.

5) 1$_Dollar_Bill - Берёт все тикеры, что есть и распределяет капитал (всегда условные 100%) между всеми тикерами.
т.е. берёт каждую сетку из СУБД, подтягивает к ней заключение Маффи + Бенкима, анализирует и выделяет %.
Потом сверяет, что бы в сумме по всем тикерам получилось 100%.

Именно после этого этапа появляются финальные аллокации. Без них должны быть заблокированы:
- `report_job`;
- финальный пользовательский PDF;
- любой user-facing report package, который выглядит как окончательный выпуск.

6) Jusetta.
как только джек получил от Бена Кима на хранение - Джузетта может, при желании, построить только `analysis_preview`/черновик.
Но финальный `report_job`, финальный пользовательский PDF и полноценный user-facing report package допускаются только после того, как завершились `Maffi` и `1$_Dollar_Bill`, а аллокации сохранены в системе.

Канонический порядок стадий такой:
- Stage 0. Jack/Data_collector готовит и хранит рыночные данные.
- Stage 1. Ben_Kim публикует analysis-only артефакты.
- Stage 2. Опционально выпускается `analysis_preview`/черновик.
- Stage 3. Maffi публикует параметры сетки.
- Stage 4. 1$_Dollar_Bill публикует финальные аллокации.
- Stage 5. Только после этого Jusetta выпускает `report_job` и пользовательский PDF.

Правила по типам артефактов:
- `analysis_result` / analysis-only output Ben_Kim:
  - промежуточный;
  - не финальный;
  - пользователю как финальный отчёт отправлять нельзя.
- `analysis_preview`:
  - промежуточный;
  - не финальный;
  - пользователю можно отправить только как явно помеченный preview/черновик.
- `grid_proposal` от Maffi:
  - промежуточный;
  - не финальный;
  - пользователю как финальный отчёт отправлять нельзя.
- `capital_allocation` от 1$_Dollar_Bill:
  - финальный upstream-артефакт для принятия решения;
  - сам по себе не является пользовательским PDF-отчётом.
- `report_job` и пользовательский PDF от Jusetta:
  - финальные;
  - могут быть отправлены пользователю;
  - должны быть заблокированы при отсутствии аллокаций.
